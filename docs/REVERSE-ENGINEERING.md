# ClearTG Analytics UA — как устроен продукт (простым языком)

Документ для тех, кто **не программист**, но должен понимать: что делает система, как она связана с рекламой и Telegram, и что уже готово.

---

## 1. Что это за продукт (одним абзацем)

**ClearTG Analytics** — это кабинет для маркетолога, который ведёт рекламу **в Telegram-канал**.

Он отвечает на вопрос: **«Человек подписался на канал — откуда он пришёл? С какой рекламы?»**

Не просто «был клик по объявлению», а **клик → подписка → отчёт** и при желании **отправка события в Meta** (чтобы рекламный кабинет видел результат).

Похоже по идее на сервисы вроде **tgtrack**, но свой код, фокус на Украину, приватность и GDPR.

---

## 2. Кто пользуется

| Кто | Зачем заходит |
|-----|----------------|
| **Владелец канала / бизнес** | Понять, какая реклама даёт подписчиков |
| **Агентство** | Вести несколько клиентов (workspace = отдельный «кабинет клиента») |
| **Таргетолог** | Подключить Meta Pixel + CAPI, смотреть конверсии |

---

## 3. Главная цепочка (сердце продукта)

Вот что должно происходить **в идеале**, когда всё подключено (бек + бот + Meta):

```
1. Маркетолог создаёт в кабинете:
   • канал (Telegram)
   • кампанию (например «Meta — весна 2026»)
   • tracking-ссылку

2. Вставляет ссылку в рекламу Meta:
   https://ваш-сайт/l/abc123?utm_source=meta&fbclid=...

3. Человек кликает по рекламе
   → система записывает клик (откуда, когда, какие метки)
   → перекидывает в Telegram (подписаться на канал)

4. Человек подписывается
   → бот в канале видит подписку
   → система связывает подписку с кликом («атрибуция»)

5. В кабинете появляются цифры:
   клики, подписчики, конверсия клик → подписка

6. Опционально: в Meta уходит событие «подписался»
   (чтобы оптимизировать рекламу)
```

**Простыми словами:** мы «склеиваем» рекламный клик и подписку в Telegram в одну историю.

## 4. Два типа ссылок

| Тип | Адрес | Когда использовать |
|-----|--------|-------------------|
| **Landing (платный трафик)** | `/l/xxxx` | Meta, Google, TikTok — в рекламе |
| **Shortlink (органика)** | `/r/xxxx` | Блогеры, посты, «просто ссылка в био» |

Разница в настройках по умолчанию: для `/l/` чаще включают **уникальную invite-ссылку на каждый клик** — так точнее понять, с какого именно клика пришёл подписчик.

## 6. Роль Telegram-бота

Бот — **не для рассылок спамом**, а для:

1. **Видеть подписки и отписки** в канале (официальный Bot API).
2. **Создавать invite-ссылки** на клик (если бот админ с правом «добавлять участников»).
3. **Lead magnet:** пользователь пишет боту «ПОГОДЖУЮСЬ» → получает файл после проверки подписки.
4. **Команды:** `/start`, `/stop` (отписаться от бота).

**Как подключить канал (когда есть бек):**

1. Создать бота в @BotFather, токен в настройки сервера.
2. Добавить бота **администратором** в канал.
3. Канал появляется в кабинете (автоматически в тестовом режиме или через API).

## 7. Атрибуция — как система понимает «откуда подписчик»

Не всегда можно сказать на 100%. Продукт честно показывает **уровень уверенности**:

| Ситуация | Насколько точно | Простое объяснение |
|----------|-----------------|-------------------|
| Подписался по **уникальной invite-ссылке с клика** | Очень точно (~98%) | «Этот клик → эта подписка» |
| Invite от кампании, но не с конкретного клика | Высоко | «Скорее всего с этой рекламы» |
| Только по времени (клик был недавно) | Средне | «Похоже на эту рекламу, но не факт» |
| Кликов не было | Органика | «Пришёл сам, не с нашей ссылки» |
| Непонятно | Низко | «Источник неизвестен» |

Окно по умолчанию: **7 дней** — если клик был неделю назад, связь слабее.

## 8. Meta и другие рекламные системы

### Meta (главное для старта)

В кабинете вводят:

- **Pixel ID**
- **Access Token** (для Conversions API)

Когда человек подписался и были нужные согласия — сервер отправляет событие в Meta **без сырого Telegram ID** (только хеш — для приватности).

### Также заложено (но не всегда нужно на старте)

- **GA4** — Google Analytics
- **Google Ads** — офлайн-конверсии
- **TikTok** — Events API

---

## 9. Приватность (почему это важно для UA / GDPR)

- IP и браузер **не хранятся как есть** — только хеш.
- Telegram ID **не уходит в Meta** как число — только salted hash.
- На клике можно передать **согласие** (analytics, marketing и т.д.).
- Есть страницы: Политика конфиденциальности, Условия, Cookies.
- Заложены запросы на удаление данных и аудит-лог.

**Честно:** автоматическое удаление всех данных по запросу и выгрузка — в схеме БД есть, **полная автоматизация ещё не везде доделана**.

## 11. Как сейчас работает деплой (важно понимать!)

### Режим А — то, что у вас сейчас на Vercel

- Только **кабинет** на Vercel.
- Данные (каналы, кампании, ссылки, Meta) — **в браузере** (localStorage).
- Вход: email + пароль из переменных Vercel.
- **Клики, бот, реальная аналитика — не работают**, пока нет бека.

Подходит: показать интерфейс, настроить «на бумаге», не мучиться с Railway.

### Режим Б — полный продукт (когда подключите бек)

- Vercel — кабинет.
- Railway/VPS — API + Postgres + Redis + бот.
- Vercel проксирует `/api`, `/l/`, `/r/` на сервер.
- Всё сохраняется в базе, бот ловит подписки, Meta получает события.

## 12. Вход в систему (три способа в коде)

| Способ | Когда | Как |
|--------|-------|-----|
| **Локальный (Vercel)** | Сейчас, без бека | Email + пароль → данные в браузере |
| **Staging** | Тест с беком | Один логин/пароль из env сервера |
| **Magic link** | Продакшн (задумано) | Ссылка на email — **в UI пока не вынесено**, только в API |

## 13. Что уже сделано vs что ещё «наполовину»

### ✅ Работает (когда бек запущен)

- Запись кликов по `/l/` и `/r/`
- Редирект / лендинг в Telegram
- Invite-ссылки на клик
- Webhook бота, подписки/отписки
- Атрибуция и отчёты
- Очередь отправки в Meta / GA4 / Google / TikTok
- Формы в кабинете: кампании, ссылки, Meta
- Lead magnet через бота
- Retention D1/D7/D30 (проверка «остался ли в канале»)

### ⚠️ Частично или только в интерфейсе

| Фича | Статус |
|------|--------|
| Magic link на странице логина | Нет кнопки, есть только API |
| Письма на email (SMTP) | Заглушка — ссылка в логах |
| Автопоявление канала без staging | Только при `STAGING_MODE=true` |
| Команда / приглашения | Страница-заглушка |
| Удаление данных по запросу | Заявка создаётся, авто-очистки нет |
| Рассылки (mailings) | Только таблицы в БД, кода нет |
| Редактирование/удаление кампаний | Только создание и просмотр |
| Локальный режим Vercel | Нет реальных кликов и бота |

## 14. Основные сущности в базе (словарь)

| Термин | Что это |
|--------|---------|
| **User** | Пользователь кабинета (email) |
| **Workspace** | «Кабинет клиента» — всё изолировано внутри |
| **Channel** | Telegram-канал |
| **Campaign** | Рекламная кампания (Meta, Google…) |
| **Tracking link** | Ссылка `/l/` или `/r/` |
| **Click event** | Факт клика по ссылке |
| **Membership event** | Подписка или отписка |
| **Attribution** | Связь «клик → подписка» |
| **Subscriber profile** | Профиль подписчика (без сырого ID в рекламе) |
| **Conversion event** | Событие для отправки в Meta и др. |
| **Invite link** | Одноразовая ссылка Telegram на клик |

## 15. Схема потока данных (картинка в тексте)

```
Реклама Meta
    │
    ▼
/l/slug  ──► ClickEvent (клик записан)
    │
    ├──► ConsentEvent (согласие с URL)
    │
    ├──► InviteLink (опционально, 1 клик = 1 invite)
    │
    ▼
Telegram канал
    │
    ▼
Бот: chat_member (подписался)
    │
    ├──► MembershipEvent
    ├──► SubscriberProfile
    ├──► Attribution (тип + уверенность)
    │
    ▼
ConversionEvent ──► очередь Redis ──► Meta CAPI / GA4 / …
    │
    ▼
Кабинет: Огляд, Звіти
```
